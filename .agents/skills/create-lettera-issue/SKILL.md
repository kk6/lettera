# Create Lettera Issue

正本文書を仕様の置き場、Issueを作業の追跡場所として扱う。既定では下書きを提示し、登録が明示された場合だけGitHubへ書き込む。

## 調査する

次の順序で、依頼に必要な範囲だけ読む。

1. リポジトリルートの`AGENTS.md`を最初に最後まで読む。
2. `README.md`と`docs/README.md`から正本文書とプロジェクト運用を確認する。
3. 対象に応じて`docs/roadmap.md`、対応する`docs/features/`、`docs/requirements-handoff.md`、`docs/architecture.md`、関連する`docs/adr/`を読む。必要なら`docs/README.md`が示す他の正本文書も読む。
4. `git status`、関連する履歴・差分・実装を読み、未コミットまたは未公開の文書を区別する。ユーザーの変更を上書きしない。
5. GitHubの既存Issueを、表記揺れ、機能名、Phase、症状を含む複数の語で検索し、open/closed双方の重複・関連Issueを確認する。
6. `.github/ISSUE_TEMPLATE/work-item.md`を読み、現在のIssue構成と記入ガイドを確認する。あわせて既存Issueと利用可能なリポジトリ情報からラベル、マイルストーンの運用を確認する。

GitHubの読み書きには、利用可能なら専用コネクタを優先し、未対応の操作だけ`gh` CLIを使う。認証情報や機密情報を表示しない。

## 分類して正本を決める

対象を一つに分類し、Issueより先に更新すべき正本がないか判断する。

- **既存Phase／既存機能仕様の実装**: 対応する`docs/features/`を正本としてリンクし、今回着手する範囲だけを切り出す。
- **新規機能**: 利用者から見える振る舞いや要件を変える規模なら、先にDraft機能仕様または既存正本文書の更新を提案する。小規模で文書化が不要ならIssueだけで扱う。
- **バグ修正**: 期待する振る舞いの正本を探す。原因が未調査なら仮説を事実にしない。修正が仕様変更を伴うなら文書更新を示す。
- **リファクタリング／保守**、**調査**、**文書更新**: 無理に機能仕様を作らず、目的、範囲、成果物、完了条件をIssueで明確にする。

docsには仕様、要件、設計判断、受け入れ条件を置く。Issueには作業の目的、背景、今回の範囲、完了条件、進捗、議論とIssue固有の差分・再現条件・制約を置く。docsを全文転載せず、詳細がある場合は短い概要とリンクにする。軽微なバグや保守のためだけにdocsを新設せず、重要な要件変更や設計判断をIssueだけで確定しない。

正本の追加・更新や未決事項によってIssueの内容が大きく変わる場合は、決めつけず、判明事項と選択肢を示してユーザーへ確認する。

## Issueを組み立てる

`.github/ISSUE_TEMPLATE/work-item.md`を構成の正本として使う。HTMLコメントと空のプレースホルダーを完成本文に残さず、タスクに不要な見出しは削除する。バグでは`バグの詳細`を残し、再現手順、実際の結果、期待する結果、影響範囲を記載する。テンプレートが見つからない、または読めない場合は勝手に別構成を作らず、その状態を報告する。

タイトルは一つの作業目的を具体的に表す。完了条件は検証可能にし、既存仕様の受け入れ条件を大量に複製しない。関連文書は可能ならGitHubで閲覧できる固定先または既定ブランチのURLにし、未コミット・未公開ならその状態を明記する。関連Issueは重複でなければ関係を短く記す。

既存ラベルやマイルストーンは、運用と内容に明確に一致する場合だけ候補にする。新しいラベルやマイルストーンは作らない。

## 登録を制御する

- 「下書きを作って」「Issue案を考えて」などは本文を提示するだけで、登録しない。
- 「Issueを作って」「Issue登録して」などGitHubへの作成が明示された場合だけ登録する。「この機能をIssueにして」は文脈に明確な登録指示がなければ下書きとして扱う。
- 大きな未決事項があれば、登録許可があっても先に確認する。
- 重複Issueが見つかったら新規登録せず、既存Issueと重複と判断した根拠を提示する。
- forward-test、検証、例示では絶対に登録しない。

## 報告する

下書きでは、分類、正本、重複調査結果、文書更新の要否を短く示し、続けてタイトル案とMarkdown本文をそのまま利用できる形で提示する。未決事項と、候補にした既存ラベル／マイルストーンがあれば分離して示す。

登録した場合はIssue番号とURL、設定した既存ラベル／マイルストーンを報告する。登録しなかった場合は、その理由（下書き依頼、重複、確認待ち）を明記する。
