# Source editor

## 状態

Draft

## 目的

利用者がアプリを起動し、プレーンテキストまたはMarkdownをすぐに入力・編集できるようにする。この機能ではReactによる入力と状態更新を学び、ファイル操作やMarkdownレンダリングとは分離する。

## 利用者から見える振る舞い

- アプリを起動すると、空の編集領域が表示される。
- 編集領域を選択し、キーボードからテキストを入力できる。
- 改行、空白、Markdown記法を含む入力が、そのまま編集領域へ反映される。
- 日本語IMEで変換中の文字列を入力・確定できる。
- 編集領域では通常のmacOSテキスト編集操作を利用できる。

## 受け入れ条件

1. 初期状態の本文は空である。
2. 英数字、日本語、記号、改行を入力できる。
3. `# 見出し`、`- リスト`、コードフェンスなどのMarkdown記法を入力しても、Source editorが内容を自動変換しない。
4. 日本語IMEの変換中に、未確定文字が重複、消失、途中確定しない。
5. 入力後にReactが再レンダリングしても、本文が失われない。
6. Undo、Redo、Copy、Cut、Paste、Select Allは、OS／ブラウザの標準的な編集操作として機能する。独自UIやメニューへの統合はこの機能の完了条件に含めない。

## 対象外

- ファイルを開く、保存する、新規作成する
- Markdownプレビュー
- Source／Splitのモード切り替え
- 文書内検索
- ツールバー、メニューバー、サイドバー
- 独自のUndo履歴
- 自動保存
- Seamless編集
- エディタライブラリの導入

## 文書状態

この機能で必要な状態は、編集中の本文だけである。ファイルパス、未保存フラグ、プレビュー用の別状態はまだ導入しない。

## 学習目標

実装後、次を自分の言葉で説明できること。

- Reactのstateが何を保持するか
- controlled textareaでは、表示値と`onChange`がどう関係するか
- state更新によって再レンダリングが起きても入力内容を保持できる理由
- 日本語IMEのcompositionと通常の入力イベントの違い
- この段階でRust Commandを必要としない理由

## 検証

### 自動確認

この段階では、フロントエンドのテストランナーを導入しない。純粋な変換ロジックやデータ損失リスクがまだなく、テスト環境の導入をReact入力の学習より優先しないためである。

既存の品質チェックを実行する。

```sh
pnpm format
pnpm lint
pnpm build
```

### 手動確認

- 英数字と句読点を入力する。
- 複数行の日本語をIMEで入力し、変換候補の選択と確定を行う。
- 見出し、リスト、引用、コードフェンスを含むMarkdownを入力する。
- Copy、Cut、Paste、Select All、Undo、Redoをキーボードで実行する。
- 長めの複数行テキストを入力し、入力内容が欠落していないことを確認する。

手動確認で問題が見つかった場合は、再現手順を残してから修正する。
