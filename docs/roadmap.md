# 開発ロードマップ

## 目的

このロードマップは納期を管理するものではなく、React、Tauri、Rustを一つずつ理解しながらLetteraを作るための学習順序を示す。各段階では、動作に加えて「どこで何が起きているか」を自分の言葉で説明できることを完了条件とする。

企画書にある初期版の全機能を、一度にMVPとして実装しない。

## Phase 0: Tauriの接続を理解する

成果：

- 初期テンプレートの`greet`を使い、ReactからRust Commandを呼ぶ流れを追える
- 引数、戻り値、非同期処理、エラーが境界をどう通るか説明できる
- `src/`と`src-tauri/`のビルドおよび起動の関係を説明できる

この段階ではエディタ機能を追加しない。既存コードを読み、必要なら小さな実験だけを行う。

## Phase 1: Source editor

成果：

- 空の文書へプレーンテキスト／Markdownを入力できる
- 日本語IMEを含む入力内容を意図せず変換しない
- Reactのstateとtextareaの関係を説明できる

仕様：[Source editor](features/001-source-editor.md)

## Phase 2: React内のMarkdownプレビュー

成果：

- SourceとSplitを切り替えられる
- Splitでは本文とプレビューが同じMarkdown状態を参照する
- Markdown解析とHTML表示に伴う安全性を説明できる

この段階ではファイル操作を扱わず、固定またはメモリ上の本文だけを使う。

## Phase 3: 最小のファイル読み込み

成果：

- ファイル選択と読み込みを、ReactからTauri／Rustの境界を通して実行できる
- 成功、キャンセル、失敗を区別できる
- 読み込んだ本文をSourceとSplitの両方で表示できる

まず一時ファイルで動作を確認し、利用者の実ファイルをテストに使わない。

## Phase 4: 別名保存

成果：

- 保存先を選び、現在の本文を書き込める
- 成功するまで保存済みとして扱わない
- キャンセルまたは失敗後も本文を保持できる
- React、Tauri Command、Rustの責務を説明できる

上書き保存より先に別名保存を実装し、パスを持たない新規文書の流れを理解する。

## Phase 5: 文書ライフサイクル

成果：

- 新規作成、開く、上書き保存、別名保存を扱える
- ファイルパスと未保存状態を管理できる
- 未保存の文書を閉じる前に確認できる
- 読み込み・保存エラーから回復できる

この段階で、状態が複雑になった場合だけ文書状態の分離方法を検討する。

## Phase 6: 基本操作の統合

成果：

- 主な操作をキーボードショートカットから実行できる
- macOSメニューバーとツールバーから同じ操作を実行できる
- 文書内検索を利用できる
- 同じ操作が複数の入口で重複実装されていない

## Phase 7: 文書構造の支援

成果：

- Markdown見出しをアウトラインとして表示できる
- アウトラインから対応箇所へ移動できる
- 最近使ったファイルから文書を再度開ける
- 参照先が移動または削除されていても、現在の文書を失わずに対処できる
- 指定フォルダーをファイルツリーとして表示できる
- ファイルツリーからテキストファイルを開ける

アウトラインとプレビューが同じMarkdown解析結果を共有すべきかは、この段階で判断する。

## Phase 8: アプリ設定

成果：

- 本文の基本フォントサイズを変更できる
- カラーモードを、システム設定に従う、ライト、ダークから選べる
- 設定変更を現在の画面へ反映できる
- アプリを再起動しても設定を維持できる
- 設定が存在しない、壊れている、または利用できない場合に既定値で起動できる

設定と最近使ったファイルはユーザー文書へ混入させず、Letteraのアプリデータとして扱う。具体的な保存ライブラリは、このPhaseへ着手するときに公式の選択肢と制約を確認して決める。

## Future: Seamless editor

SourceとSplit、文書ライフサイクル、Markdown変換が十分理解できてから検討する。既存Markdownの往復保持、IME、カーソル位置、Undo履歴への影響を調査し、独立した仕様と設計判断を作成してから着手する。

## 各Phaseの進め方

1. 機能仕様を自分で読み、不明点を言葉にする。
2. 必要なReact／Tauri／Rustの概念を調べる。
3. 小さな実験または最小の実装を自分で書く。
4. 動作と失敗時の挙動を確認する。
5. AIには説明、ヒント、コードレビューを依頼する。
6. 学んだ内容と残った疑問を、必要なら仕様または短い開発メモへ反映する。
