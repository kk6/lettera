# Lettera documentation

このディレクトリには、Letteraを学習しながら開発するために必要な文書を置く。文書を増やすこと自体を目的にせず、実装や判断に必要な情報だけを記録する。

## 文書一覧

- [企画書](proposal.md)：なぜ作るのか、誰のためのアプリか、初期版の範囲
- [Constitution](constitution.md)：開発中に守る長期的な原則
- [アーキテクチャ](architecture.md)：現在の構成、責務境界、データの流れ
- [概念データモデル](conceptual-data-model.md)：初期版で扱う主な概念と関係の見取り図
- [利用者の行動シナリオ](user-action-scenarios.md)：利用のきっかけ、行動の流れ、完了と例外
- [UI設計](ui-design.md)：初期版で利用者が触れるUI、動線、状態、ラフな配置
- [機能定義](functions.md)：初期版のワークセット、機能の入力・処理・出力、ストア系機能
- [ロードマップ](roadmap.md)：学習単位で分けた実装順
- [Source editor仕様](features/001-source-editor.md)：最初に実装する機能の振る舞い
- `adr/`：重要な設計判断と、その時点での理由（必要になったときだけ作成）

## 文書の役割

- 企画書はプロダクトの目的と範囲を表す。
- Constitutionは、個別機能を超えて長期間守る原則を表す。
- アーキテクチャ文書は、現在のコード構成に合わせて更新する。
- 利用者の行動シナリオは、具体的なUIを決める前の利用者の目的と行動を表す。
- UI設計は、UI間の動線、各UIの目的、画面上で必要になる状態を表す。
- 機能定義は、利用者の操作や外部入力に対してLetteraが行う処理と出力を表す。
- 機能仕様は、利用者から観察できる振る舞いと対象外を表す。
- ADRは、複数の選択肢から重要な判断をした時点のWhy／Whatを残す。

同じ内容を複数の文書へコピーしない。内容が変わった場合は、その情報のsource of truthとなる文書だけを更新し、必要なら参照元のリンクを直す。

## 軽量に保つためのルール

- 小さな修正や自明な実装には、新しい文書を作らない。
- 実装前に必要な範囲だけを書き、将来機能を詳細化しすぎない。
- 実装と文書が食い違った場合、どちらが正しいかを判断して同じ変更内で直す。
- 文書の維持コストが判断や学習を上回ったら、統合または削除を検討する。
